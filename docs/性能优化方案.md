# 码途系统性能优化方案（EdgeOne + COS + Turso 架构）

## 一、当前架构概览

码途采用以下部署架构：

- **部署平台**：腾讯云 EdgeOne Pages（国内 CDN 加速，替代 Vercel）
- **数据库**：Turso（SQLite 边缘数据库，免费 9GB）
- **文件存储**：腾讯云 COS 对象存储（免费 50GB + 10GB 流量/月）
- **定时任务**：cron-job.org（免费外部调度，最多 5 个 Job）
- **邮件服务**：SMTP（可选，无配置时自动降级为控制台输出）
- **全栈框架**：Next.js 15 + App Router + TypeScript

---

## 二、已识别的性能瓶颈

### 2.1 前端渲染层面

**1. layout.tsx 强制动态渲染**

`export const dynamic = 'force-dynamic'` 导致所有页面都无法利用 Next.js 的静态生成和增量静态再生（ISR）能力。每次请求都走完整的服务端渲染，增加不必要的计算开销。在 EdgeOne Pages 上，这意味着每个请求都要触发函数执行，无法命中 CDN 缓存。

**2. CSS 动画性能问题**

`globals.css` 中使用了大量 `backdrop-filter`（毛玻璃效果）和复杂的 CSS 动画：
- `backdrop-filter: blur(20px)` 在移动端和低配设备上会触发频繁的 GPU 重绘，导致滚动卡顿
- `.stagger` 类为每个子元素设置独立的动画延迟，大量 DOM 节点同时动画时会造成主线程阻塞
- `body::before` 的 SVG noise 纹理滤镜会持续占用 GPU 资源

**3. JobFeedShell 分页逻辑缺陷**

分页按钮点击后只是调用 `fetcher` 获取数据，但没有更新 SWR 的缓存状态（`mutate`），导致页面数据不会实际更新。用户点击分页后看不到内容变化。

**4. 缺少图片优化**

`next.config.ts` 只配置了 Vercel Blob 的域名，但没有启用 `next/image` 的图片优化功能。如果岗位卡片需要展示公司 Logo，图片加载会成为性能瓶颈。同时 COS 域名未配置到 `remotePatterns` 中。

**5. 组件重渲染未优化**

`JobCard`、`FilterPanel` 等组件没有使用 `React.memo`，筛选条件变化时会导致整个列表重新渲染。

### 2.2 数据库层面

**1. 缺少数据库索引**

`schema.prisma` 中没有任何索引定义。随着岗位数据量增长（目标 5000+ 活跃岗位），以下查询会进行全表扫描：
- `JobPosting.status = 'active'` — 首页默认筛选
- `JobPosting.lastRefreshedAt` — 排序字段
- `JobPosting.city` — 城市筛选
- `JobPosting.companyType` — 单位类型筛选
- `JobPosting.jobDirection` — 岗位方向筛选
- `MatchRecord.userId + score` — 推荐 Feed 查询
- `SavedJob.userId + jobId` — 收藏查询

**2. JSON 字段反复解析**

`techStack`、`matchedSkills`、`missingSkills` 等字段以 JSON 字符串存储，每次查询都需要 `JSON.parse()`，增加 CPU 开销。在 `/api/jobs` 中，20 条岗位记录每条都要解析一次 `techStack`。

**3. 查询返回冗余字段**

`/api/jobs` 的 `findMany` 返回了完整的 `JobPosting` 模型，包括 `jdFullText` 大文本字段。列表页只需要展示摘要信息，不需要完整的 JD 文本。

### 2.3 服务端层面

**1. Cron 任务超时风险**

`/api/cron/crawl` 将多个抓取源合并在一个端点中：
- 51job 抓取（国内源，20秒超时）
- V2EX 抓取（海外源，15秒超时，经常失败）
- 种子数据补充

如果 V2EX 超时，整个 Cron 任务会等待超时后才返回。cron-job.org 的 HTTP 请求也有超时限制，可能导致任务被标记为失败。

**2. 匹配引擎性能问题**

`services/matcher/index.ts` 中的 `matchAll()` 函数：
- 对每个用户，遍历所有活跃岗位进行匹配
- 使用 `upsert` 逐条写入数据库，没有批量操作
- 如果 100 个用户 × 5000 个岗位 = 50 万次循环，每次循环都有数据库操作

**3. 简历解析同步阻塞**

`/api/resume/upload` 中，上传文件后同步调用 `parseResume()` 进行解析。如果 PDF 文件较大或解析耗时，API 响应时间会很长，可能导致 EdgeOne Pages 函数超时。

**4. 缺少 API 响应缓存**

所有 API 响应都没有设置 `Cache-Control` 头，浏览器和 CDN 无法缓存数据。岗位列表这种读多写少的数据应该可以被缓存。

### 2.4 部署架构层面

**1. EdgeOne Pages 函数冷启动**

EdgeOne Pages 的 Serverless 函数有冷启动时间。`layout.tsx` 中的 `getUser()` 每次请求都会查询数据库，即使用户信息没有变化，增加了冷启动后的延迟。

**2. Turso 网络延迟**

Turso 是边缘数据库，虽然比中心化数据库快，但仍有 20-50ms 的网络往返延迟。对于需要多次查询的 API（如 `/api/jobs` 先 count 再 findMany，再查收藏和匹配），延迟会累积。

**3. COS 图片加载未优化**

如果未来使用 COS 存储公司 Logo，直接返回原始 URL 没有进行图片压缩、格式转换（WebP）等优化。

## 三、优化方案

### 3.1 前端渲染优化

#### 3.1.1 移除 layout.tsx 的强制动态渲染

将 `export const dynamic = 'force-dynamic'` 从 `layout.tsx` 移除，改为在需要动态数据的具体页面中按需使用。对于首页岗位列表，可以通过 ISR 或客户端数据获取来实现动态更新。

```typescript
// layout.tsx — 移除这行
// export const dynamic = 'force-dynamic';

// 改为在需要的地方使用
// page.tsx 中：
export const revalidate = 60; // ISR 每 60 秒重新生成
```

#### 3.1.2 优化 CSS 动画性能

1. **减少 backdrop-filter 使用**：将 `.glass` 和 `.glass-card` 的 `backdrop-filter` 降级为纯色背景 + 轻微透明度，仅在桌面端保留毛玻璃效果：

```css
.glass-card {
  background: hsl(var(--surface) / 0.95);
  border: 1px solid hsl(var(--border) / 0.6);
  border-radius: var(--radius-xl);
  transition: transform 0.2s ease, box-shadow 0.2s ease;
}

@media (min-width: 1024px) {
  .glass-card {
    background: hsl(var(--surface) / 0.7);
    backdrop-filter: blur(12px) saturate(150%);
  }
}
```

2. **简化 stagger 动画**：将逐个元素的动画延迟改为批量动画，减少 GPU 占用：

```css
.stagger > * {
  animation: slide-up 0.4s ease-out forwards;
  animation-delay: calc(min(var(--index, 0), 5) * 0.05s);
}
```

3. **移除 body 的 grain 纹理**：`body::before` 的 SVG noise 滤镜会持续占用 GPU 资源，建议改为静态 PNG 背景或仅在首屏使用。

#### 3.1.3 修复分页逻辑

使用 SWR 的 `mutate` 函数正确更新分页数据：

```typescript
const { data, isLoading, mutate } = useSWR(url, fetcher, { keepPreviousData: true });

function goToPage(page: number) {
  const params = new URLSearchParams(queryParams);
  params.set('page', String(page));
  mutate(fetcher(`/api/jobs?${params.toString()}`));
}
```

#### 3.1.4 启用 React.memo 和 useMemo

对 `JobCard` 和 `FilterPanel` 组件添加 `React.memo`，避免不必要的重渲染：

```typescript
export const JobCard = React.memo(function JobCard({ job }: JobCardProps) {
  // ...
});
```

#### 3.1.5 图片懒加载与优化

配置 COS 域名到 `next.config.ts`，使用 `next/image` 优化图片加载：

```typescript
const nextConfig = {
  images: {
    remotePatterns: [
      { protocol: 'https', hostname: '**.blob.vercel-storage.com' },
      { protocol: 'https', hostname: '**.cos.**.*.myqcloud.com' },
      { protocol: 'https', hostname: '**.myqcloud.com' },
    ],
  },
};
```

### 3.2 数据库优化

#### 3.2.1 添加数据库索引

在 `schema.prisma` 中添加复合索引，显著提升查询性能：

```prisma
model JobPosting {
  // ... 现有字段

  // 复合索引：状态 + 刷新时间（首页默认查询）
  @@index([status, lastRefreshedAt(sort: Desc)])
  // 复合索引：状态 + 城市（城市筛选）
  @@index([status, city])
  // 复合索引：状态 + 单位类型（单位类型筛选）
  @@index([status, companyType])
  // 复合索引：状态 + 岗位方向（方向筛选）
  @@index([status, jobDirection])
  // 复合索引：状态 + 薪资范围（薪资筛选）
  @@index([status, salaryMin, salaryMax])
  // 全文搜索索引（SQLite FTS5，需要额外配置）
  @@index([title])
  @@index([companyName])
}

model MatchRecord {
  // ... 现有字段

  // 复合索引：用户 + 分数（推荐 Feed 查询）
  @@index([userId, score(sort: Desc)])
  // 索引：匹配时间（清理过期匹配）
  @@index([matchedAt])
}

model SavedJob {
  // ... 现有字段

  // 索引：用户 + 创建时间（收藏列表排序）
  @@index([userId, createdAt(sort: Desc)])
}

model User {
  // ... 现有字段

  // 索引：邮箱（登录查询）
  @@index([email])
}
```

执行迁移：

```bash
npx prisma migrate dev --name add_performance_indexes
```

#### 3.2.2 预计算 freshness 字段

将 `freshness` 计算从查询时移到写入时，避免每次查询都进行时间比较：

```prisma
model JobPosting {
  // ... 现有字段
  freshness String? @default("today") // today | 7d | 30d | expired
}
```

在抓取入库时计算并存储：

```typescript
function computeFreshness(lastRefreshedAt: Date): string {
  const days = (Date.now() - lastRefreshedAt.getTime()) / (24 * 60 * 60 * 1000);
  if (days < 1) return 'today';
  if (days < 7) return '7d';
  if (days < 30) return '30d';
  return 'expired';
}
```

#### 3.2.3 批量更新过期岗位状态

Cron 任务中批量更新过期岗位，而不是逐条检查：

```typescript
// 在 /api/cron/crawl/route.ts 中
await prisma.jobPosting.updateMany({
  where: {
    status: 'active',
    lastRefreshedAt: { lt: new Date(Date.now() - 30 * 24 * 60 * 60 * 1000) }
  },
  data: { status: 'expired', freshness: 'expired' }
});
```

### 3.4 服务端优化

#### 3.4.1 Cron 任务分拆与容错

将抓取任务按来源分拆，避免单个源超时导致全部失败：

```json
// vercel.json（或 cron-job.org 配置）
{
  "crons": [
    { "path": "/api/cron/crawl/seed", "schedule": "0 */6 * * *" },
    { "path": "/api/cron/crawl/51job", "schedule": "0 * * * *" },
    { "path": "/api/cron/crawl/v2ex", "schedule": "*/30 * * * *" },
    { "path": "/api/cron/match", "schedule": "0 2 * * *" },
    { "path": "/api/cron/mail", "schedule": "0 9 * * *" }
  ]
}
```

每个抓取任务独立设置超时：

```typescript
async function fetchWithTimeout(url: string, timeoutMs = 15000) {
  const controller = new AbortController();
  const timeout = setTimeout(() => controller.abort(), timeoutMs);
  try {
    const res = await fetch(url, { signal: controller.signal });
    return res;
  } finally {
    clearTimeout(timeout);
  }
}
```

#### 3.4.2 匹配引擎批量优化

将匹配结果的逐条 `upsert` 改为批量操作：

```typescript
// 使用 Prisma 的 createMany + updateMany 替代逐条 upsert
const matchData = results.map(r => ({
  userId,
  jobId: r.jobId,
  score: r.score,
  matchedSkills: JSON.stringify(r.matchedSkills),
  missingSkills: JSON.stringify(r.missingSkills),
}));

// 先尝试批量创建，忽略冲突
await prisma.matchRecord.createMany({
  data: matchData,
  skipDuplicates: true,
});

// 再批量更新已存在的记录
for (const batch of chunkArray(matchData, 100)) {
  await Promise.all(batch.map(d => 
    prisma.matchRecord.updateMany({
      where: { userId: d.userId, jobId: d.jobId },
      data: d,
    })
  ));
}
```

#### 3.4.3 简历解析异步化

将同步解析改为异步队列模式，上传后立即返回，后台解析：

```typescript
// /api/resume/upload/route.ts
export async function POST(req: Request) {
  // ... 上传文件

  // 不再同步解析，只保存文件并标记为 pending
  const resume = await prisma.resume.create({
    data: {
      userId: user.id,
      fileName: file.name,
      filePath,
      parseStatus: 'pending',
    },
  });

  // 触发异步解析（可以调用另一个 API 或放入队列）
  // 前端轮询解析状态
  return NextResponse.json({
    resume: { id: resume.id, fileName: resume.fileName, parseStatus: 'pending' },
  });
}

// 新增 /api/resume/parse-async/route.ts
export async function POST(req: Request) {
  const { resumeId } = await req.json();
  const resume = await prisma.resume.findUnique({ where: { id: resumeId } });
  if (!resume) return NextResponse.json({ error: '不存在' }, { status: 404 });

  // 标记解析中
  await prisma.resume.update({
    where: { id: resumeId },
    data: { parseStatus: 'parsing' },
  });

  try {
    const result = await parseResume(resume.filePath);
    // ... 更新画像
    await prisma.resume.update({
      where: { id: resumeId },
      data: { parseStatus: 'done', parsedText: JSON.stringify(result) },
    });
    return NextResponse.json({ status: 'done', result });
  } catch (err) {
    await prisma.resume.update({
      where: { id: resumeId },
      data: { parseStatus: 'failed', parseError: String(err) },
    });
    return NextResponse.json({ status: 'failed', error: String(err) });
  }
}
```

前端轮询解析状态：

```typescript
// 前端上传后轮询
async function pollParseStatus(resumeId: string) {
  for (let i = 0; i < 30; i++) {
    const res = await fetch(`/api/resume/profile?resumeId=${resumeId}`);
    const data = await res.json();
    if (data.parseStatus === 'done') return data;
    if (data.parseStatus === 'failed') throw new Error(data.parseError);

  }
}
```

### 3.5 部署架构优化

#### 3.5.1 启用 EdgeOne Pages 缓存策略

EdgeOne Pages 支持通过响应头控制 CDN 缓存。对静态资源和公共 API 添加缓存头：

```typescript
// 静态资源（已自动处理）
// API 响应手动添加
return NextResponse.json(data, {
  headers: {
    'Cache-Control': 'public, max-age=60, s-maxage=300',
  },
});
```

#### 3.5.2 使用 EdgeOne 的图片处理功能

腾讯云 COS 支持图片处理参数（如缩放、格式转换），在返回图片 URL 时添加处理参数：

```typescript
// 在 lib/blob.ts 中，返回带图片处理的 URL
export function getProcessedImageUrl(rawUrl: string, width: number, height: number): string {
  if (!isCosUrl(rawUrl)) return rawUrl;
  // COS 图片处理参数
  return `${rawUrl}?imageView2/1/w/${width}/h/${height}/format/webp`;
}
```

## 四、优化实施优先级

| 优先级 | 优化项 | 预期收益 | 实施难度 | 涉及文件 |
|--------|--------|----------|----------|----------|
| P0 | 添加数据库索引 | 查询速度提升 10-100 倍 | 低 | `prisma/schema.prisma` |
| P0 | 修复分页逻辑 | 功能可用性 | 低 | `components/jobs/JobFeedShell.tsx` |
| P0 | 精简 API 返回字段 | 减少网络传输 30-50% | 低 | `app/api/jobs/route.ts` |
| P1 | 预计算 freshness 字段 | 减少查询时 CPU 计算 | 低 | `prisma/schema.prisma`, `services/crawler/index.ts` |

## 六、EdgeOne Pages 特定优化

### 6.1 函数执行时间优化

EdgeOne Pages 的函数执行时间有限制（通常 10-30 秒），以下操作需要特别注意：

1. **简历解析**：同步解析大 PDF 可能超时，必须改为异步模式
2. **全量匹配**：`matchAll()` 遍历所有用户和岗位，可能超时，建议改为分批处理或仅匹配新入库岗位
3. **V2EX 抓取**：海外源经常超时，建议缩短超时时间或降低抓取频率

### 6.2 国内 CDN 优化

EdgeOne Pages 使用腾讯云 CDN，对国内访问有天然优势：

1. **静态资源缓存**：Next.js 构建的静态资源（JS/CSS）会自动被 CDN 缓存
2. **API 缓存**：通过 `Cache-Control` 头可以让 CDN 边缘节点缓存 API 响应
3. **COS 就近访问**：COS 存储桶选择离用户最近的区域（如广州），配合 CDN 加速

### 6.3 cron-job.org 调度优化

cron-job.org 免费版限制：
- 最多 5 个 Job
- 最小执行间隔 1 分钟
- HTTP 请求超时约 30 秒

优化建议：
1. 将抓取任务按来源分拆（51job、V2EX、种子数据）
2. 设置合理的超时时间（10-15 秒）
3. 在 cron-job.org 中配置失败通知（邮件），及时发现抓取异常

## 七、长期优化方向

1. **搜索功能**：当岗位数据量达到 1 万+ 时，考虑接入 Algolia 或 Meilisearch 实现全文搜索和毫秒级筛选。

2. **数据分片**：如果用户量增长，考虑将 Turso 数据库按地区分片，或使用 EdgeOne 的 Edge Config 缓存热点数据。

3. **图片 CDN**：如果添加公司 Logo，使用腾讯云 COS 的图片处理功能自动优化格式和尺寸。

4. **Web Workers**：将简历解析的文本提取逻辑移到 Web Worker，避免阻塞主线程。

5. **边缘缓存**：使用 EdgeOne 的 KV 存储或缓存 API，将热门岗位列表缓存到边缘节点，减少回源请求。
