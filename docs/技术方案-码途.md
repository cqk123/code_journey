# 「码途」计算机工作机会信息平台 — 技术方案

> **版本**：2.0
> **日期**：2026-06-14
> **设计原则**：一个技术栈贯穿前后端，架构从简，MVP优先，免费部署

---

## 1. 技术选型

### 1.1 选型核心思路

前后端**共用同一语言（TypeScript）+ 同一框架（Next.js）**，一个项目跑全栈，零跨语言成本。部署到 **Vercel**，所有第三方服务（数据库、文件存储、定时任务）全部用免费额度的云方案，**¥0 上线**。

### 1.2 技术栈一览

| 层级 | 选型 | 理由 |
|------|------|------|
| **全栈框架** | Next.js 15 (App Router) | 前端页面 + API 路由全在一个项目里，原生适配 Vercel |
| **语言** | TypeScript | JS 生态通用语言，前后端都用它 |
| **数据库** | Turso（SQLite 边缘数据库）+ Prisma ORM | SQLite 协议兼容，零运维，免费 9GB；Prisma 自动生成类型 |
| **文件存储** | Vercel Blob | 免费 10GB，S3 兼容 API，专门存简历文件 |
| **样式** | Tailwind CSS | 不写 CSS 文件，直接在 JSX 里用语义类名 |
| **认证** | 自建 JWT（jsonwebtoken） | 最简方案，不引入第三方登录 |
| **抓取** | cheerio | 轻量 HTML 解析 |
| **定时任务** | Vercel Cron Jobs | 在 `vercel.json` 声明，自动按 cron 表达式调度 |
| **简历解析** | pdf-parse + mammoth | 轻量 PDF/Word 文本提取 |
| **邮件** | nodemailer | 发送验证码、匹配通知 |

### 1.3 全免费额度

| 服务 | 免费额度 | 够用吗 |
|------|------|:--:|
| Vercel | 100GB 带宽/月，6000分钟构建/月 | ✅ |
| Turso | 500 个数据库，9GB 存储，10亿行读取/月 | ✅ 远够 |
| Vercel Blob | 10GB 存储，100GB 带宽/月 | ✅ |
| Vercel Cron | Hobby Plan 可配定时任务 | ✅ |

### 1.4 为什么不用

| 被排除的 | 排除原因 |
|---------|---------|
| 前后端分离（如 React + Express） | 多一个项目、多一套部署、多一套联调 |
| MySQL / PostgreSQL | 需要单独安装数据库服务 |
| 本地 SQLite 文件 | Serverless 环境无持久化磁盘，需换成 Turso |
| Redis | MVP 阶段无高并发缓存需求 |
| 微服务 / 消息队列 | 功能简单，单体应用足够 |
| 第三方登录（GitHub/Google） | 邮箱注册最简单，不依赖外部服务配置 |
| Python 爬虫 | 既然前后端都用 JS，爬虫也用 JS 保持栈统一 |
| node-cron | Next.js API Routes 上 cron 随进程启停不可靠，换成 Vercel Cron |

## 3. 数据库设计

### 3.1 Prisma Schema（Turso 连接）

```prisma
datasource db {
  provider = "sqlite"
  url      = env("TURSO_DATABASE_URL")
}

generator client {
  provider        = "prisma-client-js"
  previewFeatures = ["driverAdapters"]
}

// ========== 用户体系 ==========

model User {
  id            String    @id @default(cuid())
  email         String    @unique
  passwordHash  String
  isVerified    Boolean   @default(false)
  verifyCode    String?
  verifyCodeExp DateTime?
  createdAt     DateTime  @default(now())

  profile       JobSeekerProfile?
  resumes       Resume[]
  savedJobs     SavedJob[]
  filterPresets FilterPreset[]
  matchRecords  MatchRecord[]
}

model JobSeekerProfile {
  id              String  @id @default(cuid())
  userId          String  @unique
  skillTags       String  @default("[]")  // JSON数组
  workYears       Float?
  educationJson   String  @default("{}")
  expectCities    String  @default("[]")
  expectSalaryMin Int?
  expectSalaryMax Int?
  expectJobTypes  String  @default("[]")
  expectOrgTypes  String  @default("[]")
  updatedAt       DateTime @updatedAt

  user User @relation(fields: [userId], references: [id])
}

model Resume {
  id          String   @id @default(cuid())
  userId      String
  fileName    String
  filePath    String              // Vercel Blob URL
  parsedText  String?
  parseStatus String   @default("pending")  // pending | parsing | done | failed
  parseError  String?
  createdAt   DateTime @default(now())

  user User @relation(fields: [userId], references: [id])
}

// ========== 岗位体系 ==========

model JobPosting {
  id               String   @id @default(cuid())
  title            String
  companyName      String
  companyType      String?
  city             String?
  salaryMin        Int?
  salaryMax        Int?
  salarySource     String?           // company_labeled | system_inferred
  jobDirection     String?
  techStack        String   @default("[]")
  jdFullText       String?
  sourceUrl        String?
  sourcePlatform   String            // boss | liepin | lagou | v2ex | corporate
  publishedAt      DateTime?
  lastRefreshedAt  DateTime?
  status           String   @default("active")  // active | expired | closed
  createdAt        DateTime @default(now())
  updatedAt        DateTime @updatedAt

  savedJobs    SavedJob[]
  matchRecords MatchRecord[]
}

// ========== 收藏与匹配 ==========

model SavedJob {
  id        String   @id @default(cuid())
  userId    String
  jobId     String
  note      String?
  createdAt DateTime @default(now())

  user User        @relation(fields: [userId], references: [id])
  job  JobPosting  @relation(fields: [jobId], references: [id])

  @@unique([userId, jobId])
}

model MatchRecord {
  id              String   @id @default(cuid())
  userId          String
  jobId           String
  score           Int                  // 0-100
  matchedSkills   String   @default("[]")
  missingSkills   String   @default("[]")
  matchedAt       DateTime @default(now())

  user User       @relation(fields: [userId], references: [id])
  job  JobPosting @relation(fields: [jobId], references: [id])

  @@unique([userId, jobId])
}

// ========== 筛选预设 ==========

model FilterPreset {
  id          String   @id @default(cuid())
  userId      String
  name        String
  filterJson  String
  createdAt   DateTime @default(now())
  lastUsedAt  DateTime?

  user User @relation(fields: [userId], references: [id])
}
```

> 注意：Prisma 模型定义与本地 SQLite 版本**完全相同**，只是 datasource 从本地文件路径改为了 Turso 环境变量。因为 Turso 就是 SQLite 协议，Prisma 的 SQLite provider 完全兼容。

### 3.2 ER 关系速览

```
User ──1:1──→ JobSeekerProfile
User ──1:N──→ Resume
User ──1:N──→ SavedJob ──N:1──→ JobPosting
User ──1:N──→ MatchRecord ──N:1──→ JobPosting
User ──1:N──→ FilterPreset
```

## 5. 核心模块设计

### 5.1 数据库连接 (`lib/db.ts`)

Prisma Client 通过 `@prisma/adapter-libsql` 适配器连接到 Turso 边缘数据库：

```typescript
import { PrismaClient } from '@prisma/client'
import { PrismaLibSQL } from '@prisma/adapter-libsql'
import { createClient } from '@libsql/client'

const libsql = createClient({
  url: process.env.TURSO_DATABASE_URL!,
  authToken: process.env.TURSO_AUTH_TOKEN!,
})

const adapter = new PrismaLibSQL(libsql)

const globalForPrisma = globalThis as unknown as { prisma: PrismaClient }

export const prisma = globalForPrisma.prisma || new PrismaClient({ adapter })

if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = prisma
```

### 5.2 简历文件存储 (`lib/blob.ts`)

简历上传后存入 Vercel Blob，数据库中 `filePath` 存的是 Blob 公网 URL，后续解析时直接 fetch 下载：

```typescript
import { put, del } from '@vercel/blob';

// 上传简历到 Blob
export async function uploadResumeBlob(file: File, userId: string) {
  const blob = await put(`resumes/${userId}/${Date.now()}-${file.name}`, file, {
    access: 'public',
    token: process.env.BLOB_READ_WRITE_TOKEN,
  });
  return blob.url;
}

// 删除简历 Blob（用户删除简历时调用）
export async function deleteResumeBlob(url: string) {
  await del(url, { token: process.env.BLOB_READ_WRITE_TOKEN });
}
```

### 5.3 岗位抓取模块 (`services/crawler/`)

**设计思路**：一个调度器 + 多个源适配器，所有抓取逻辑统一返回 `JobRawData` 结构，由去重层统一入库。

```
Vercel Cron 定时触发 GET /api/cron/crawl
                    │
                    ▼
            CrawlScheduler
          ┌─────┼─────┐
          ▼     ▼     ▼
      BossZhipin  Liepin  Lagou  V2EX
          └─────┬─────┘
                ▼
           DedupFilter    ← 标题+公司+城市去重
                ▼
           Prisma.create   ← 统一入库到 Turso
```

**抓取策略**：

| 来源 | 频率 | 备注 |
|------|:--:|------|
| Boss直聘 / 猎聘 / 拉勾 | 每1小时 | 热门平台，高频抓取 |
| V2EX 酷工作 | 每30分钟 | 数据量小，高频率无害 |

**爬虫反封策略（从简）**：
- 每次请求间隔 3~5 秒随机延迟
- 公共 User-Agent，不伪造
- 单线程顺序抓取，不做并发
- 仅抓取列表页摘要信息（标题、公司、城市、薪资），不请求每个岗位详情页——**大幅降低被封风险**

**去重逻辑**：
```
去重 key = hash(标题前20字 + 公司名 + 城市)
```
- 同一个 key 的岗位只保留最新一条
- 已入库的岗位再次抓取到时，更新 `lastRefreshedAt`

**Vercel Cron 限制说明**：
- Hobby 方案下每次函数最大执行时间 60 秒
- 如果抓取 4 个源耗时超过 60 秒，需要分拆为多个 cron 端点，或合并为一个 cron 内部调度

### 5.4 简历解析模块 (`services/resume/`)

**设计思路**：两步走——先从 Blob URL 下载文件提取文本，再基于规则提取关键词。

```
Vercel Blob URL
  │
  ▼
fetch(filePath) → ArrayBuffer
  │
  ▼
pdf-parse / mammoth  ← 提取纯文本
  │
  ▼
技术栈词库匹配        ← 100+ 主流技术关键词白名单
  │
  ▼
正则提取工作年限      ← 匹配"X年"、"X年经验" 等模式
  │
  ▼
技能画像 (skillTags[])  ← 返回前端，用户可手动修正
```

**技术栈词库（白名单，摘录）**：

```
语言: Java, Python, Go, Rust, C++, TypeScript, JavaScript, Kotlin, Swift
框架: Spring Boot, Django, Flask, Express, Next.js, React, Vue, Angular, Gin, FastAPI
数据: MySQL, PostgreSQL, MongoDB, Redis, Elasticsearch, Kafka, RabbitMQ
容器: Docker, Kubernetes
AI: TensorFlow, PyTorch, LangChain, LLM, RAG, Agent, Transformer
云: AWS, Azure, GCP, 阿里云
工具: Git, Nginx, Jenkins, GitHub Actions, Linux
```

**错误处理**：
- 加密PDF → `parseError: "加密文件无法解析"`
- 图片型PDF → `parseError: "简历解析遇到问题"`
- 文件过大(>10MB) → 上传阶段直接拒绝

### 5.5 匹配引擎 (`services/matcher/`)

**设计思路**：纯计算，不依赖外部AI。基于技能重合率、经验匹配、意向匹配三个维度加权打分。

```
匹配得分 = 技能得分 × 0.6 + 经验得分 × 0.2 + 意向得分 × 0.2

技能得分 = (用户技能 ∩ 岗位要求技能) / 岗位要求技能数量 × 100
经验得分 = 1 - |用户工作年限 - 岗位要求年限| / max(用户年限, 岗位年限) × 100
  如果岗位未注明经验要求，默认60分
意向得分 = 城市匹配(0.4) + 薪资匹配(0.3) + 岗位方向匹配(0.15) + 单位类型匹配(0.15)
```

**触发时机**：

| 时机 | 范围 | 说明 |
|------|------|------|
| 用户上传简历 / 修改意向 | 该用户 × 所有活跃岗位 | 实时（API 调用即可） |
| 新岗位入库 | 该岗位 × 所有已激活用户 | 实时（抓取后立即执行） |
| 每日凌晨 | 所有用户 × 所有活跃岗位 | Vercel Cron 批量刷新 |

### 5.6 通知模块 (`services/notifications/`)

| 通知类型 | 触发条件 | 推送方式 |
|---------|---------|:--:|
| 新岗位匹配 | 新入库岗位匹配度 ≥ 80 | 站内通知（下次请求通知页时返回） |
| 每日汇总 | 每天 9:00（用户选择了"每日"频率） | 邮件 |
| 每周汇总 | 每周一 9:00（用户选择了"每周"频率） | 邮件 |

**实现方式**：不引入 WebSocket。站内通知由用户请求 `/api/notifications` 时查询未读 MatchRecord 返回。邮件由 Vercel Cron 定时触发 `/api/cron/mail` 汇总发送。

## 7. 认证方案

**极简 JWT 方案**，cookie 携带，无需第三方：

```
注册流程:
  邮箱+密码 → 生成6位验证码 → 存User表 → 发邮件 →
  用户输入验证码 → 校验 → isVerified=true → 签发JWT → httpOnly cookie

登录流程:
  邮箱+密码 → 校验 → 签发JWT → httpOnly cookie

每次请求:
  cookie中的JWT → 中间件校验 → 注入user到请求上下文
```

**JWT 结构**：

```typescript
type JwtPayload = {
  userId: string;
  email: string;
};

const token = jwt.sign(payload, process.env.JWT_SECRET!, { expiresIn: '7d' });

// 中间件（lib/auth.ts）
export async function getUser(req: NextRequest) {
  const token = req.cookies.get('token')?.value;
  if (!token) return null;
  try {
    const payload = jwt.verify(token, process.env.JWT_SECRET!);
    return await prisma.user.findUnique({ where: { id: payload.userId } });
  } catch {
    return null;
  }
}
```
