# 「码途」计算机工作机会信息平台 产品需求文档

> **版本**：1.0
> **日期**：2026-06-14
> **作者**：AI辅助生成

---

## 1. 产品概述

### 1.1 一句话定位

为1~5年经验的计算机开发者提供的跨平台岗位聚合与智能匹配信息平台。

### 1.2 核心价值主张

为1~5年经验的计算机开发者，提供一个跨平台聚合 + 简历智能匹配的岗位信息平台。不同于传统招聘网站信息分散、筛选粗糙，我们实现全网岗位一站聚合、简历驱动的精准匹配、以及时间×城市×单位类型×岗位方向×薪资的五维组合筛选。

## 2. 背景与目标

### 2.1 市场背景

2026年中国IT/计算机招聘市场正处于**结构性调整期**。前程无忧《2026企业招聘行情与用人趋势洞察》报告显示，AI加速渗透背景下，人工智能、大数据等领域的高端技术人才需求持续增长，但传统CRUD类岗位在缩减。Boss直聘、猎聘、拉勾三大平台2026年联合发布的《AI技术人才趋势报告》指出，不懂Agent+Skill生态的工程师正被大厂JD逐步排除。计算机就业已进入"高端缺人、低端淘汰"的分化阶段。

现有问题：
- **信息分散**：计算机岗位散落在Boss直聘、猎聘、拉勾、前程无忧、企业官网、技术社区（V2EX/掘金/牛客）等十余个渠道，求职者需逐个平台切换搜索
- **筛选能力弱**：主流平台对技术栈（如"React+Go"、"Python+大模型"）的精细化筛选支持不足
- **信息时效性差**：已关闭的职位仍显示在列，导致求职者无效投递
- **薪资不透明**：大量岗位标注"薪资面议"，求职者难以横向比较

### 2.2 产品目标

- **O**：成为计算机开发者找工作的第一站——「打开码途，不再切换平台」
  - KR1：上线3个月内日活跃岗位数达5,000+
  - KR2：用户留存率（次日）≥ 40%
  - KR3：简历上传率（注册用户中）≥ 70%

### 2.3 成功指标

- **北极星指标**：周活用户中「使用筛选功能并收藏岗位」的用户数
- **辅助指标**：
  - 推荐Feed点击率（CTR）
  - 用户日均浏览岗位数
  - 筛选条件保存率

## 3. 用户画像

### 画像1：小李 — 1~3年初级程序员

- **基本信息**：25岁，本科/硕士计算机相关专业，毕业后进入第一家公司，做过1-2个项目，掌握1-2门主流语言和框架（如Java+Spring、React），技术深度有限，对市场薪资行情不太了解
- **典型场景**：每天下班后刷Boss直聘、拉勾、牛客网，海投50份简历回复寥寥，不清楚自己到底能匹配什么水平的岗位
- **核心任务**：JTBD-1（全网聚合）、JTBD-2（简历匹配推送）、JTBD-3（多维筛选）
- **当前痛点**：简历石沉大海，不知道问题出在哪；在不同平台重复搜索同样的关键词，效率极低

### 画像2：老王 — 3~5年中高级程序员

- **基本信息**：30岁，在某个技术方向（如后端Go、大数据、AI工程化）有较深的积累，带过小团队或独立负责过核心模块
- **典型场景**：有跳槽想法但不会海投，只看特定方向、特定薪资区间的岗位。每月花大量时间在多个平台筛选、排除不匹配岗位，同时关注行业薪资走势
- **核心任务**：JTBD-1（全网聚合）、JTBD-2（智能匹配）、JTBD-3（多维筛选）、JTBD-4（薪资透明）
- **当前痛点**：每天花1小时筛选，80%的岗位跟自己技术栈不匹配；薪资信息不透明，难以横向比较

## 4. 需求分析

### 4.1 JTBD清单

|----|---------|:--:|------------|
| JTBD-1 | 当我每天有碎片时间刷岗位时，我想要**一个地方就能看到全网的计算机岗位汇总**，这样我就不用在七八个平台之间来回切换 | P0 | 小李、老王 |

| JTBD-4 | 当我看到感兴趣的岗位时，我想要看到**真实的薪资范围和岗位时效性**，这样我就能横向比较并优先投递还在活跃招聘的岗位 | P0 | 小李、老王 |

## 6. 产品架构

### 6.1 功能模块划分

| 模块ID | 模块名称 | 包含功能 | 职责描述 |
|:--:|---------|---------|---------|
| M-0 | 基础模块 | F-A, F-B, F-C | 账户体系、用户设置、通知中心等基础设施 |
| M-1 | 岗位发现模块 | F-1, F-4, F-5, F-6, F-7, F-8 | 岗位数据的聚合展示、多维筛选、个性化推送、薪资与时效呈现 |
| M-2 | 个人画像模块 | F-2, F-3, F-10 | 简历上传解析、技能画像生成、求职意向配置、人岗匹配引擎 |
| M-3 | 我的岗位模块 | F-9 | 岗位收藏管理、多岗位并排对比 |

### 6.2 模块关系

```
M-2（个人画像）──→ M-1（岗位发现）  技能画像+求职意向驱动个性化推送和筛选默认值
M-1（岗位发现）──→ M-3（我的岗位）  用户从发现页收藏岗位进入管理页
M-0（基础模块）──→ M-2（个人画像）  用户登录后绑定简历与画像
M-0（基础模块）──→ M-1（岗位发现）  通知体系向发现页推送消息
```

### 7.2 状态流转

#### 岗位状态机

```
        抓取入库 → 活跃 → [>30天未刷新] → 过期下沉
                    │
                    └── [来源端已删除/404] → 已关闭（隐藏但保留记录）
```

| 从 | 到 | 触发条件 | 触发方 |
|----|-----|---------|:--:|
| 抓取入库 | 活跃 | 首次抓取成功入库 | 系统 |
| 活跃 | 活跃（刷新） | 同一岗位再次抓取到，更新刷新时间 | 系统 |
| 活跃 | 过期下沉 | `当前时间 - 最近刷新时间 > 30天` | 系统定时任务 |
| 活跃 | 已关闭 | 来源URL返回404或明确下架 | 系统（抓取检测） |

#### 简历状态机

```
        待上传 → 解析中 → 已解析 → [重新上传] → 解析中 → 已解析（覆盖旧版）
                   │
                   └── 解析失败 → 展示失败原因 + 重试入口
```

### 7.3 异常路径

| 场景 | 触发条件 | 用户看到什么 | 可执行操作 |
|------|---------|------------|----------|
| 岗位数据为空 | 平台首日上线，岗位尚未收录 | 引导图：「岗位正在收录中，上传简历后我们会第一时间通知你」→ 引导上传简历 | 上传简历 / 设置求职意向 |
| 筛选无结果 | 五个筛选条件过于苛刻 | 空结果提示：「没有找到匹配的岗位，试试放宽条件？」→ 列出建议放宽的维度 | 一键放宽 / 清除所有筛选 |
| 简历解析失败 | 格式异常/加密PDF/图片型PDF | 失败提示：「简历解析遇到问题，请确认上传的是PDF或Word格式文件」 | 重新上传 / 手动填写技能 |
| 匹配无高分 | 技能画像与当前岗位库差距较大 | 推荐Feed展示：「暂未发现高匹配岗位，以下是最接近的岗位」按匹配度降序 | 查看最接近岗位 / 调整求职意向 |
| 网络错误 | 请求超时或服务器异常 | 页面级Toast：「网络异常，请稍后重试」+ 重试按钮 | 手动重试 |
| 未登录点击收藏 | 未登录状态点击收藏按钮 | 弹出登录/注册引导弹窗，收藏操作暂存 | 登录后自动执行收藏 |
| 列表超量 | 筛选结果超500条 | 分页加载，每页20条，底部「加载更多」 | 滚动加载 / 进一步筛选 |
| 删除简历 | 在简历管理页点击删除 | 二次确认弹窗：「删除后技能画像将被清空，匹配推荐将不再生效。确定删除吗？」 | 确认删除 / 取消 |

## 8. 功能详细设计

### M-1：岗位发现模块

#### F-1：多源岗位抓取聚合

- **功能描述**：系统从多个招聘渠道自动抓取计算机开发相关岗位信息，经格式化清洗后统一入库展示
- **来源JTBD**：JTBD-1
- **输入**：各招聘平台的公开岗位页面、企业官网招聘页、技术社区招聘板块
- **输出**：统一格式的岗位卡片列表（岗位Feed）
- **抓取源**：
  - 综合平台：Boss直聘、猎聘、拉勾、前程无忧
  - 企业官网：主流科技公司招聘页（周期性抓取）
  - 技术社区：V2EX「酷工作」板块、掘金招聘、牛客网
- **业务规则**：
  - 同一岗位（标题+公司+城市+发布时间完全一致）自动去重
  - 抓取频率：热门平台每1小时，企业官网每6小时
  - 仅收录明确标注为计算机/技术开发类的岗位
- **异常处理**：
  - 抓取失败：记录日志，下个周期重试；连续失败3次则标记该源为异常
  - 反爬拦截：逐步降低频率 + 更换User-Agent
- **数据来源**：网络抓取 + 定时任务入库
- **约束条件**：需遵守各来源的robots协议和使用条款

#### F-4：个性化岗位推送

- **功能描述**：基于用户技能画像和求职意向，在推荐Feed展示高匹配度岗位，并通过通知触达
- **来源JTBD**：JTBD-2
- **输入**：用户技能画像（F-2产出）+ 求职意向（F-10产出）
- **输出**：推荐Feed列表 + 通知消息
- **业务规则**：
  - 匹配度 ≥ 60分的岗位进入推荐Feed
  - 匹配度 ≥ 80分的岗位触发通知推送
  - 推荐Feed默认按匹配度降序排列
  - 已收藏岗位不再重复出现在推荐Feed首位
- **关联功能**：依赖 F-2（技能画像）、F-3（匹配引擎）、F-10（求职意向）、F-C（通知）
- **数据来源**：MatchRecord表 + JobPosting表联合查询

#### F-5：多维度组合筛选

- **功能描述**：用户可同时按五个维度组合过滤岗位，实时刷新结果
- **来源JTBD**：JTBD-3
- **输入**：用户在筛选面板的勾选/拖动操作
- **输出**：过滤后的岗位卡片列表
- **筛选维度与选项**：

| 维度 | 选项 | 交互方式 |
|------|------|:--:|
| 时间 | 近24小时 / 近7天 / 近30天 / 全部 | 单选按钮组 |
| 城市 | 北京/上海/深圳/杭州/成都/广州/南京/武汉/苏州/西安/其他 | 多选标签 |
| 单位类型 | 大厂(BAT/TMD等) / 中型企业(200-2000人) / 创业公司 / 国企&央企 / 外企 | 多选标签 |
| 岗位方向 | 后端 / 前端 / 移动端 / AI&算法 / 数据 / 测试&运维 / 全栈 / 其他 | 多选标签 |
| 薪资范围 | 月薪 ¥5k - ¥80k+ | 双滑块 |

- **交互说明**：筛选面板默认展开在Feed左侧（桌面端）/ 顶部可收起的抽屉（移动端）。每勾选/调整一项，列表即时刷新。显示「共 N 个岗位」
- **业务规则**：
  - 不设筛选条件时 = 展示全部活跃岗位
  - 多选项之间为「或」关系，跨维度之间为「且」关系
  - 薪资范围只对「企业标注」和「系统推断」（置信度≥60%）的岗位生效
- **异常处理**：
  - 筛选结果为空：展示空状态提示 + 建议放宽的条件 + 一键清除按钮

#### F-6：筛选条件保存

- **功能描述**：用户可将当前筛选条件命名保存，后续在「我的订阅」中一键加载
- **来源JTBD**：JTBD-3
- **业务规则**：每个用户最多保存10组筛选条件预设
- **关联功能**：依赖 F-5（筛选操作产生条件JSON）

#### F-7：薪资提取与展示

- **功能描述**：从岗位JD中智能提取薪资信息，标注来源类型，支持月薪/年薪切换查看
- **来源JTBD**：JTBD-4
- **输出**：薪资范围 + 来源标注
- **薪资来源标注**：

| 标注类型 | 含义 | 展示样式 |
|---------|------|------|
| 🏷️ 企业标注 | JD中明确写出薪资数字 | 绿色标签，可信度高 |
| 🤖 系统推断 | 从文本中提取到薪资相关表述 | 黄色标签，标注置信度 |
| — | 未提取到薪资信息 | 显示「暂未标注」，不参与薪资筛选 |

- **业务规则**：
  - 支持月薪/年薪一键切换（年薪 = 月薪 × 12 + 年终参考值）
  - 「薪资面议」类岗位标记为「暂未标注」

#### F-8：岗位时效标注

- **功能描述**：展示每个岗位的发布时间和活跃状态，过期岗位自动下沉
- **来源JTBD**：JTBD-4
- **输出**：岗位卡片上的时间标签 + 列表排序影响
- **展示规则**：

| 时效 | 展示标签 | 排序权重 |
|------|------|:--:|
| <24小时 | 「今天」+ 绿色NEW标签 | 提升 |
| 1-7天 | 「x天前」 | 正常 |
| 8-30天 | 「x天前」 | 正常 |
| >30天 | 「x天前」+ 橙色标签 | 下沉，不出现在默认Feed |
| 已关闭 | 不在Feed中展示 | — |

### M-2：个人画像模块

#### F-2：简历上传与解析

- **功能描述**：用户上传PDF或Word格式简历，系统自动提取技术栈、工作年限、项目经验等关键信息
- **来源JTBD**：JTBD-2
- **输入**：PDF/Word简历文件（≤10MB）
- **输出**：结构化技能画像
  - 技术栈标签列表（如：Java, Spring Boot, MySQL, Redis, Docker）
  - 工作年限
  - 教育背景
  - 最近项目经验摘要
- **交互说明**：
  - 上传区域支持拖拽和点击选择
  - 解析过程显示进度动画（通常3-10秒）
  - 解析完成后展示提取结果，用户可手动修改标签、补充遗漏
- **异常处理**：
  - 格式不支持：「请上传PDF或Word格式文件」
  - 文件过大：「文件大小不能超过10MB」
  - 解析失败：「解析遇到问题，请检查文件是否损坏或加密」
  - 提取信息过少：「简历信息较少，建议手动补充关键技能」
- **关联功能**：F-3 智能匹配依赖此功能产出

#### F-3：智能匹配与评分

- **功能描述**：将用户技能画像与岗位JD做双向匹配，输出匹配度评分和技能差距分析
- **来源JTBD**：JTBD-2
- **输入**：用户技能画像 + 岗位技术栈要求
- **输出**：MatchRecord（匹配得分 + 技能匹配JSON + 技能差距[]）
- **匹配规则**：
  - 技能匹配（权重60%）：用户会的技能 ∩ 岗位要求的技能
  - 经验匹配（权重20%）：工作年限 vs 岗位要求年限
  - 意向匹配（权重20%）：城市、薪资、岗位方向、单位类型
- **展示方式**：
  - 岗位卡片顶部：圆形匹配度徽章（颜色编码：80+ 绿色 / 60-79 黄色 / <60 灰色）
  - 岗位详情页：匹配明细面板 —— 「你匹配了：Java ✓、Spring Boot ✓、MySQL ✓」「你缺少：Kubernetes、系统设计经验」
- **业务规则**：
  - 匹配得分每天刷新一次，或用户更新画像时立即刷新
  - 新岗位入库时自动跑一次全量用户匹配
- **关联功能**：依赖 F-2、F-10；被 F-4 消费

#### F-10：求职意向设置

- **功能描述**：用户手动配置期望的城市、薪资范围、岗位类型、单位类型
- **来源JTBD**：JTBD-2
- **业务规则**：
  - 期望城市默认为简历中的最近工作城市
  - 期望薪资默认基于简历工作年限推算的市场中位数 [建议值，待确认]
  - 修改任意意向 → 触发重新匹配（F-3）
  - 意向设置可在无简历时独立使用（此时仅作用于F-5筛选默认值）
- **关联功能**：F-3（匹配引擎输入）、F-5（筛选默认值）

| 术语 | 说明 |
